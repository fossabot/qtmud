qtmud package¶

qtmud is an early-alpha MUD engine. Using an event subscription model, qtmud handles the interactions between Things which been built using builders to create arbitrarily complex game objects.

Right now that doesn’t mean very much - Things are relatively simple, and the only real quality that qtmud can apply is to turn a generic thing into a client. This basic framework is used by mudlibs to create full games, however. Check out Starhopper for one example.

Module contents¶

The main qtmud module houses some constants, the Thing class, and methods for start()ing and tick()ing the MUD engine.

The file where pickled client accounts should be stored.

qtmud.IP4_ADDRESS = ('0.0.0.0', 5787)¶

Your IPv4 address is expected to be a tuple of (‘address’, port), where address is a string and port is an integer. Set your address to ‘localhost’ for testing and development, and ‘0.0.0.0’ for production/gameplay.

qtmud.IP6_ADDRESS = ('2606:a000:4701:b900:5604:830e:39f3:12f8', 5788, 0, 0)¶

Your IPv6 address is expected to be a four-element tuple, where the first element is your IPv6 address, the second is qtmud’s bound IPv6 port, and I don’t know what the last two elements do, to be quite honest.

qtmud.load()[source]¶

Most importantly, puts every function from qtmud.subscriptions and every class from qtmud.services into subscribers and active_services, respectively.

After populating subscribers and active_services, calls the start() method for each class in active_services, then attempts to load MUDLIB, if one is specified, and finally calls load_client_accounts().

Submodules¶

qtmud’s submodules have been broken up into a few rough groups:

  • Builders make or modify game objects
  • Cmds are player-entered commands
  • Parser isn’t currently used, but houses NLTK functions
  • services are constantly-running game services
  • Subscriptions are events qtmud.tick() fires to

qtmud.subscriptions module¶

Subscriptions are methods which handle the interaction and manipulation of things.

Every method in this module is added to qtmud.subscribers when qtmud.start(). Calls to these methods which have been scheduled as events qtmud.events> will be called when qtmud.tick() is called.

Glossary¶

These are the terms which have a specific meaning under qtmud

MUD
Multi-User Dimension. A more historic term for MMORPG, MUDs were some of the earliest multiplayer games. Wikipedia has a pretty comprehensive article on them.
MUD client
A specialized client for playing MUDs. (duh)
MUD engine
Also known as drivers, MUD engines are game engines meant for presenting MUDs. There have traditionally been two main classes of MUD engine: DIKU and LPC. DIKU MUDs came first, and used database-driven logic to present a simplistic hack-and-slash environment to the player. LPC MUDs used object-oriented programming techniques to render more detailed game worlds to the player. qtmud follows in the tradition of the latter.
mudlib
short for Multi-User Dimension Library, a mudlib is a game written for a MUD engine to run. From qtmud’s perspective, a mudlib is a Python package which is heavily reliant on the qtmud module.
