qtMUD API¶

This is the complete API for qtMUD

qtmud module¶

The main qtmud module houses some constants, the Thing class, and methods for start()ing and tick()ing the MUD engine.

class qtmud.Thing(**kwargs)[source]¶

Bases: object

Most objects clients interact with are Things

Created with new_thing(), things are objects with a few attributes added on, mostly for enabling in-game reference of the objects.

name¶

Properly-cased full name of a thing Any name a thing is given is also added to Thing.nouns, with the old name being removed. Same for adjectives.

Warning

This has some wonkiness, in that “Eric Baez” can be referred to as “Eric Baez” or “Baez” but not “Eric”. “Eric Thing” would work, though.

nouns = None¶

nouns represent lower-case nouns which may be used to reference the thing.

update(attributes)[source]¶

Update multiple attributes of Thing at once.

Example:
>>> foo = Thing()
>>> foo.update({'name': 'eric'})
>>> foo.name
eric

Warning

Does not properly recognize setters

qtmud.load()[source]¶

Most importantly, puts every function from qtmud.subscriptions and every class from qtmud.services into subscribers and active_services, respectively.

After populating subscribers and active_services, calls the start() method for each class in active_services, then attempts to load MUDLIB, if one is specified, and finally calls load_client_accounts().

qtmud.cmds.help(client, topic='', *, H=False, h=False, domain='')[source]¶

The command for receiving in-game help. Searches through commands, subscribers, and active_services and returns either a list of available matching helpfiles, or the docstring of the single match.

param client:The client issuing the help command. (That’d be you.) This isn’t part of the command you enter.
param H:Shows the client this docstring.
param h:Shows the client a brief help.
param topic:None by default, otherwise the name of the command/service/subscriber you’re looking to get help with.
param domain:Optionally limit where you’re looking for help to either cmds, subscribers, or services.

For example, you might enter help talker to learn more about qtMUD’s talker system. You would be prompted to enter either help --domain=cmds talker to learn about the in-game talker command or help --domain=services talker to learn about the Talker service`.

qtmud.subscriptions module¶

Subscriptions are methods which handle the interaction and manipulation of things.

Every method in this module is added to qtmud.subscribers when qtmud.start(). Calls to these methods which have been scheduled as events will be called when qtmud.tick() is called.

Glossary¶

MUD
Multi-User Dimension. A more historic term for MMORPG, MUDs were some of the earliest multiplayer games. Wikipedia has a pretty comprehensive article on them.
MUD client
A specialized client for playing MUDs. (duh)
MUD engine
Also known as drivers, MUD engines are game engines meant for presenting MUDs. There have traditionally been two main classes of MUD engine: DIKU and LPC. DIKU MUDs came first, and used database-driven logic to present a simplistic hack-and-slash environment to the player. LPC MUDs used object-oriented programming techniques to render more detailed game worlds to the player. qtmud follows in the tradition of the latter.
mudlib
short for Multi-User Dimension Library, a mudlib is a game written for a MUD engine to run. From qtmud’s perspective, a mudlib is a Python package which is heavily reliant on the qtmud module.
